# TLS Implementation Bug Analysis

This directory contains a regenerated, layered analysis of TLS-library bug candidates created on or after `2026-01-01T00:00:00Z`.

## Files

- `tls_bug_analysis_pipeline.py`: collection and metadata triage script.
- `auto_candidates.json`: broad automatic bug-like candidate set, not confirmed bugs.
- `auto_rfc_probable_candidates.json`: automatic RFC 5246/8446-probable candidate set from heuristic filtering.
- `manual_triage_auto_rfc_probable.json`: deterministic metadata triage generated by the script.
- `analysis_counts.json`: counts from deterministic script triage.
- `subagent_triage_counts.json`: conservative subagent review counts used for paper-facing discussion.
- `paper_summary.md`: compact Korean summary in the requested paper-oriented form.
- `raw/`: raw API responses retained as evidence.
- `review_chunks/`: per-library chunks used for subagent review.

## Method

The analysis is intentionally layered.

1. The script collects a broad candidate set from GitHub and GitLab using `created_at >= 2026-01-01T00:00:00Z`.
2. The script labels candidates as bug-like using labels and text signals, then removes duplicate and explicit not-a-bug signals from the automatic RFC-probable set.
3. The script narrows to automatic RFC-probable candidates using TLS/RFC keywords plus normative or failure keywords. This `automatic RFC-probable` label is a heuristic filter, not the same as the later triaged `RFC-probable` class.
4. The automatic RFC-probable set is reviewed in per-library chunks by subagents. This produces the conservative paper-facing counts in `subagent_triage_counts.json`.
5. Per-item publication claims still require final source audit against maintainer comments, linked PR/MR, commits, release notes, and advisories.

The regenerated run produced `2,421` automatic bug-like candidates and `212` automatic RFC-probable candidates. These replace the stale `2,326/231` working numbers from the earlier draft.
