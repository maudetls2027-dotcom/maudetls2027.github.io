## Cross-Paper Pattern

Across the papers, the recurring TLS testing assumption is not just "field mutation". It separates into modeled message deviations, trace deviations, negotiation/path deviations, and concrete implementation-facing deviations.

1. Message field/value deviations: mutate modeled fields such as protocol versions, cipher suites, groups, alert descriptions, and modeled length fields.
2. Message structure/component deviations: add, remove, or empty modeled message components such as extensions, lists, optional fields, and certificate-list-level contents.
3. Trace-level deviations: omit, insert, repeat, reorder, or send wrong-role messages against the expected TLS state machine.
4. Negotiation/path/state deviations: drive alternate modeled negotiation paths or state features through fields, extensions, object state, guard relaxation, and scenario composition.
5. Concrete byte/cryptographic payload deviations: manipulate concrete record bytes, raw content, fragmentation, record/message byte truncation, padding, MAC/tag/ciphertext/PRF/signature/RSA ciphertext bytes, or cryptographic response oracles.
6. Transport/timing deviations: close/reset transport connections, omit `close_notify`, truncate transport/application delivery, trigger timeout behavior, measure timing, or classify alert/TCP/timeout/timing responses.

The current Maude behavior deviation DSL covers modeled C1-C2 deviations, C4 negotiation/path/state deviations, and part of C3 trace deviations. C5 and C6 remain the boundary where concrete byte constructors, transport/timing control, or oracle-observation model extensions are required. Certificate-internal validation is intentionally outside this category table; certificate-list-level message contents remain covered by C2.
